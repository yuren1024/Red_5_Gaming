# 红五五家记分工具设计文档（V1）

## 1. 目标与范围

本文档用于定义一个 `5 人专用` 的红五五家记分小工具第一版设计方案。

第一版仅支持百科中的 `一般情况` 结算，不包含：

- `压庄`
- `独打`
- `逃跑`
- `直拚直搭`

第一版必须支持：

- 固定 5 名玩家，默认编号为 `A / B / C / D / E`
- 玩家可自定义改名
- 每局录入 `庄家`、`伴家`、`闲家抓分`
- 根据百科规则自动结算 5 人本局分数
- 自动累计总分
- 展示总分排行榜
- 统计每个人的胜率
- 统计每个人的叫庄概率
- 支持历史记录导出

规则来源采用用户指定的百科版本：

- [红五五家_百度百科](https://baike.baidu.com/item/%E7%BA%A2%E4%BA%94%E4%BA%94%E5%AE%B6/5864225)

## 2. 产品定位

该工具是一个 `本地记分台账工具`，重点不是对局本身，而是对局后的快速结算、累计和复盘。

建议做成 `本地网页应用`：

- 打开即用
- 不依赖服务器
- 后续方便扩展 `压庄 / 独打 / 导入导出 / 统计图表`

#### A. 顶部标题区

用途：

## 4. 核心业务规则

### 4.1 参与角色

每局固定为 5 人：

- `1 名庄家`
- `1 名伴家`
- `3 名闲家`

### 4.2 结算输入

第一版仅需要用户录入：

- 谁是庄家
- 谁是伴家
- 闲家本局抓分

系统自动推出：

- 哪三人是闲家
- 本局胜方
- 五名玩家各自结算分

### 4.3 胜负判定

按照百科描述，并结合其详细记分表，第一版采用以下判定口径：

- `闲家抓分 < 120`：庄家一方获胜
- `闲家抓分 >= 120`：闲家一方获胜

### 4.4 一般情况的结算分配

每局会产出三类结算值：

- `dealerScore`：庄家本局得分
- `partnerScore`：伴家本局得分
- `idleScorePerPlayer`：每个闲家本局得分

然后映射到 5 个玩家：

- 庄家拿 `dealerScore`
- 伴家拿 `partnerScore`
- 3 个闲家各拿 `idleScorePerPlayer`

## 5. 一般情况结算规则表

第一版建议将规则实现为 `显式查表`，而不是直接写复杂公式。原因是百科在高分段存在特殊写法，查表更稳，也更方便后续验证。

### 5.2 高分段说明

根据百科中的一般情况表，`225 分` 之后：

- `伴家得分` 固定为 `-9`
- `每个闲家得分` 每增加 5 分就 `+1`
- `庄家得分` 每增加 5 分就额外多扣 `-3`

因此：

- `230 -> -21 / -9 / +10`
- `235 -> -24 / -9 / +11`
- ...
- `300 -> -63 / -9 / +24`

第一版应优先以 `查表结果` 作为唯一结算依据。

### 5.3 输入校验规则

为了避免录入错误，建议添加以下校验：

- 分数必须在 `0 ~ 300`
- 分数必须为 `5` 的倍数
- 庄家和伴家必须是不同玩家
- 每局必须恰好有 `1` 名庄家和 `1` 名伴家

若校验失败，前端给出明确提示，不允许保存。

## 6. 数据结构设计

### 6.1 玩家数据

```json
{
  "players": [
    { "id": "A", "name": "A" },
    { "id": "B", "name": "B" },
    { "id": "C", "name": "C" },
    { "id": "D", "name": "D" },
    { "id": "E", "name": "E" }
  ]
}
```

设计说明：

- `id` 固定，不随改名变化
- `name` 可编辑
- 所有历史记录都使用 `id` 关联，避免改名后历史丢失

### 6.2 单局记录数据

```json
{
  "id": "round-20260410-001",
  "roundNo": 1,
  "mode": "normal",
  "dealerId": "A",
  "partnerId": "C",
  "idlePlayerIds": ["B", "D", "E"],
  "idleScore": 145,
  "winnerSide": "idle",
  "settlement": {
    "A": -2,
    "B": 1,
    "C": -1,
    "D": 1,
    "E": 1
  },
  "ruleSnapshot": {
    "dealerScore": -2,
    "partnerScore": -1,
    "idleScorePerPlayer": 1
  },
  "note": "",
  "createdAt": "2026-04-10T10:00:00.000Z",
  "updatedAt": "2026-04-10T10:00:00.000Z"
}
```

字段说明：

- `mode` 第一版固定为 `normal`
- `idlePlayerIds` 由系统自动推导
- `settlement` 保存该局 5 人最终分值
- `ruleSnapshot` 保存该局命中的规则结果，便于调试和导出
- `note` 先预留，后续可写扣底、特殊说明

### 6.3 应用状态数据

```json
{
  "appVersion": "v1",
  "ruleVersion": "baike-normal-only",
  "players": [
    { "id": "A", "name": "A" },
    { "id": "B", "name": "B" },
    { "id": "C", "name": "C" },
    { "id": "D", "name": "D" },
    { "id": "E", "name": "E" }

```

设计原则：

- `players` 和 `rounds` 即为主数据
- `排行榜 / 总分 / 胜率 / 叫庄概率` 全部应由前端实时计算
- 不建议把统计结果持久化，避免编辑历史后数据不一致

### 6.4 统计结果结构

该结构建议运行时计算，不单独持久化。

```json

  "name": "张三",

  "roundCount": 12,

  "dealerCount": 3,

  "partnerCount": 2,
```

## 7. 统计口径设计

### 7.1 总分排行榜

定义：

- 每位玩家的 `totalScore = 所有局 settlement[playerId] 之和`

排序：

- 按 `totalScore` 从高到低排序
- 若同分，按 `winRate` 排
- 若仍相同，按 `dealerCount` 排

### 7.2 胜率

定义：

- `胜率 = 获胜局数 / 总参局数`

第一版中每位玩家默认参与全部记录，因此：

- `总参局数 = 总局数`

胜负判定规则：

- 如果该玩家当局属于 `庄家方（庄 + 伴）`，且本局胜方为庄家方，则该玩家记 1 胜
- 如果该玩家当局属于 `闲家方`，且本局胜方为闲家方，则该玩家记 1 胜

示例：

- A 是庄，C 是伴，B/D/E 是闲
- 若闲家抓分 `145`
- 则 `B/D/E` 本局记胜，`A/C` 本局记负

### 7.3 叫庄概率

由于第一版只记录 `谁最终是庄`，不记录其他玩家是否曾尝试叫庄，因此这里的“叫庄概率”在产品中建议明确定义为：

- `叫庄概率（展示名可写为“叫庄率”） = 成为庄家的局数 / 总局数`

也就是统计 `最终坐庄频率`。

备注：

- 如果后续真的要计算“主动叫庄成功率”，则必须新增事件数据，例如 `谁翻 2`、`谁亮主`、`谁尝试过叫庄`
- 第一版不具备这类原始数据，因此不能做“叫庄成功率”

### 7.4 其他可顺手统计的指标

虽然第一版不一定首屏展示，但建议在数据层保留可计算能力：

- `dealerCount`：当庄次数
- `partnerCount`：当伴次数
- `idleCount`：当闲次数
- `bestRoundScore`：单局最高得分
- `worstRoundScore`：单局最低得分

## 8. 结算流程设计

### 8.1 单局保存流程

```text
1. 用户选择庄家
2. 用户选择伴家
3. 系统自动推出 3 名闲家
4. 用户输入闲家抓分
5. 系统做合法性校验
6. 根据规则表查得 dealerScore / partnerScore / idleScorePerPlayer
7. 生成 settlement
8. 写入 rounds
9. 重算排行榜、总分、胜率、叫庄率
10. 刷新历史记录表
```

### 8.2 结算伪代码

```js
function settleNormalRound(players, dealerId, partnerId, idleScore) {
  const rule = lookupNormalRule(idleScore);
  const idlePlayerIds = players
    .map((p) => p.id)
    .filter((id) => id !== dealerId && id !== partnerId);

  const settlement = {};

  for (const player of players) {
    if (player.id === dealerId) settlement[player.id] = rule.dealerScore;
    else if (player.id === partnerId) settlement[player.id] = rule.partnerScore;
    else settlement[player.id] = rule.idleScorePerPlayer;
  }

  return {
    winnerSide: rule.winnerSide,
    idlePlayerIds,
    settlement,
    ruleSnapshot: rule
  };
}
```

## 9. 导出设计

第一版建议支持两种导出：

- `CSV`
- `JSON`

### 9.1 CSV 导出

适合：

- 发给牌友看
- 用 Excel 继续统计

建议列：

```text
局号,庄家,伴家,闲家1,闲家2,闲家3,闲家抓分,胜方,A,B,C,D,E,创建时间
```

可选补充：

- 第二个 CSV 导出 `排行榜快照`

示例列：

```text
玩家ID,玩家名,总分,胜率,叫庄率,庄家次数,伴家次数,闲家次数
```

### 9.2 JSON 导出

适合：

- 备份
- 后续导入恢复
- 将来切换到新版本继续兼容

建议结构：

```json
{
  "appVersion": "v1",
  "ruleVersion": "baike-normal-only",
  "exportedAt": "2026-04-10T10:00:00.000Z",
  "players": [],
  "rounds": []
}
```

## 10. 本地存储建议

若未来实现为前端页面，建议使用浏览器 `localStorage`：

- `hwwj.players`
- `hwwj.rounds`
- `hwwj.settings`

或统一为：

- `hwwj.appState`

推荐统一存一个对象，便于版本升级迁移。

## 11. MVP 开发顺序建议

如果后续开始编码，建议按下面顺序推进：

1. 先完成 `玩家改名 + 单局录入 + 规则查表 + 本局结算`
2. 再完成 `历史记录列表 + 删除/编辑`
3. 再完成 `总分排行榜 + 胜率 + 叫庄率`
4. 最后补 `CSV / JSON 导出`

这样能保证最早拿到一个可用版本。

## 12. 第一版边界说明

第一版明确不做：

- 压庄
- 独打
- 逃跑
- 直拚直搭
- 扣底翻倍单独录入
- 牌局过程记录
- 多桌管理

但数据结构已经为后续扩展预留：

- `mode`
- `note`
- `ruleVersion`

## 13. 后续扩展预留

后续加入 `压庄 / 独打 / 逃跑` 时，建议保持同一套页面，仅扩展：

- `mode` 选择器
- 不同模式下的角色输入逻辑
- 对应规则查表

也就是说，当前 V1 的设计不会阻碍第二版继续扩展。
