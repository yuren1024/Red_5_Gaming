# 红五五家记分工具设计文档（V2）

## 1. 目标与范围

本文档用于定义一个 `5 人专用` 的红五五家记分小工具当前版本设计方案。

当前版本支持百科中的以下结算场景：

- `一般情况`
- `独打`
- `逃跑`

当前版本暂不包含：

- `压庄`
- `直拚直搭`

当前版本必须支持：

- 固定 5 名玩家，默认编号为 `A / B / C / D / E`
- 玩家可自定义改名
- 每局录入 `模式`、`结束方式` 与对应字段
- 根据百科规则自动结算 5 人本局分数
- 自动累计总分
- 展示总分排行榜
- 统计每个人的胜率
- 统计每个人的叫庄概率
- 支持历史记录导出

当前版本的具体录入组合为：

- `一般情况 + 抓分结算`
- `独打 + 抓分结算`
- `一般情况 + 逃跑`
- `独打 + 逃跑`

规则来源采用用户指定的百科版本：

- [红五五家_百度百科](https://baike.baidu.com/item/%E7%BA%A2%E4%BA%94%E4%BA%94%E5%AE%B6/5864225)

## 2. 产品定位

该工具是一个 `本地记分台账工具`，重点不是对局本身，而是对局后的快速结算、累计和复盘。

已实现为 `本地网页应用`：

- 打开即用
- 不依赖服务器
- 当前使用浏览器本地存储，无需服务器
- 当前结构已为后续扩展 `压庄 / 直拚直搭 / 导入导出 / 统计图表` 预留

## 4. 核心业务规则

### 4.1 参与角色

每局固定为 5 人，但根据模式不同，角色结构不同：

- `一般情况`：`1 名庄家 + 1 名伴家 + 3 名闲家`
- `独打`：`1 名庄家 + 4 名闲家`

### 4.2 结算输入

当前版本根据模式和结束方式录入：

- 必填：谁是庄家、当前模式、结束方式
- `一般情况 + 抓分结算`：额外录入伴家、闲家抓分
- `独打 + 抓分结算`：额外录入闲家抓分
- `任意模式 + 逃跑`：额外录入逃跑玩家

系统自动推出：

- 哪些人是闲家
- 本局胜方或获胜玩家集合
- 五名玩家各自结算分

### 4.3 胜负判定

按照百科描述，并结合其详细记分表，当前版本采用以下判定口径：

- `抓分结算` 时：
  - `闲家抓分 < 120`：庄家一方获胜
  - `闲家抓分 >= 120`：闲家一方获胜
- `逃跑结算` 时：
  - 逃跑玩家判负
  - 其余四名玩家判胜

### 4.4 一般情况的结算分配

每局会产出三类结算值：

- `dealerScore`：庄家本局得分
- `partnerScore`：伴家本局得分
- `idleScorePerPlayer`：每个闲家本局得分

然后映射到 5 个玩家：

- 庄家拿 `dealerScore`
- 伴家拿 `partnerScore`
- 3 个闲家各拿 `idleScorePerPlayer`

### 4.5 独打的结算分配

独打情况下没有伴家，每局会产出两类结算值：

- `dealerScore`：庄家本局得分
- `idleScorePerPlayer`：每个闲家本局得分

然后映射到 5 个玩家：

- 庄家拿 `dealerScore`
- 4 个闲家各拿 `idleScorePerPlayer`

### 4.6 逃跑的结算分配

逃跑不走抓分查表，而是固定罚分：

- `独打逃跑`：逃跑玩家 `-200`，其余四家各 `+25`
- `非独打逃跑`：逃跑玩家 `-80`，其余四家各 `+10`

实现上建议把逃跑建模为 `结束方式`，而不是单独的新模式：

- `mode = normal | solo`
- `finishType = score | escape`

### 5.2 高分段说明

根据百科中的一般情况表，`225 分` 之后：

- `伴家得分` 固定为 `-9`
- `每个闲家得分` 每增加 5 分就 `+1`
- `庄家得分` 每增加 5 分就额外多扣 `-3`

因此：

- `230 -> -21 / -9 / +10`
- `235 -> -24 / -9 / +11`
- ...
- `300 -> -63 / -9 / +24`

当前版本应优先以 `查表结果` 作为唯一结算依据。

### 5.3 输入校验规则

为了避免录入错误，建议添加以下校验：

- 分数必须在 `0 ~ 300`
- 分数必须为 `5` 的倍数
- 庄家和伴家必须是不同玩家
- 每局必须恰好有 `1` 名庄家和 `1` 名伴家

若校验失败，前端给出明确提示，不允许保存。

## 6. 数据结构设计

### 6.1 玩家数据

```json
{
  "players": [
    { "id": "A", "name": "A" },
    { "id": "B", "name": "B" },
    { "id": "C", "name": "C" },
    { "id": "D", "name": "D" },
    { "id": "E", "name": "E" }
  ]
}
```

设计说明：

- `id` 固定，不随改名变化
- `name` 可编辑
- 所有历史记录都使用 `id` 关联，避免改名后历史丢失

### 6.2 单局记录数据

```json
{
  "id": "round-20260410-001",
  "roundNo": 1,
  "mode": "normal",
  "finishType": "score",
  "dealerId": "A",
  "partnerId": "C",
  "idlePlayerIds": ["B", "D", "E"],
  "idleScore": 145,
  "escapePlayerId": null,
  "winnerSide": "idle",
  "winnerPlayerIds": ["B", "D", "E"],
  "settlement": {
    "A": -2,
    "B": 1,
    "C": -1,
    "D": 1,
    "E": 1
  },
  "ruleSnapshot": {
    "dealerScore": -2,
    "partnerScore": -1,
    "idleScorePerPlayer": 1
  },
  "note": "",
  "createdAt": "2026-04-10T10:00:00.000Z",
  "updatedAt": "2026-04-10T10:00:00.000Z"
}
```

字段说明：

- `mode` 取值为 `normal | solo`
- `finishType` 取值为 `score | escape`
- `partnerId` 在独打时为 `null`
- `idlePlayerIds` 由系统自动推导
- `idleScore` 在逃跑时为 `null`
- `escapePlayerId` 仅在逃跑时有值
- `winnerPlayerIds` 用于统一表达获胜玩家，避免统计逻辑只依赖庄闲两方
- `settlement` 保存该局 5 人最终分值
- `ruleSnapshot` 保存该局命中的规则结果，便于调试和导出
- `note` 先预留，后续可写扣底、特殊说明

### 6.3 应用状态数据

```json
{
  "appVersion": "v2",
  "ruleVersion": "baike-normal-solo-escape",
  "players": [
    { "id": "A", "name": "A" },
    { "id": "B", "name": "B" },
    { "id": "C", "name": "C" },
    { "id": "D", "name": "D" },
    { "id": "E", "name": "E" }
  ],
  "rounds": []
}
```

设计原则：

- `players` 和 `rounds` 即为主数据
- `排行榜 / 总分 / 胜率 / 叫庄概率` 全部应由前端实时计算
- 不建议把统计结果持久化，避免编辑历史后数据不一致
- 旧版历史记录加载时应自动补齐 `finishType / escapePlayerId / winnerPlayerIds` 等缺失字段

### 6.4 统计结果结构

该结构建议运行时计算，不单独持久化。

```json
{
  "playerId": "A",
  "name": "张三",
  "totalScore": 27,
  "roundCount": 12,
  "winCount": 7,
  "winRate": 0.5833,
  "dealerCount": 3,
  "dealerRate": 0.25,
  "partnerCount": 2,
  "idleCount": 7
}
```

## 7. 统计口径设计

### 7.1 总分排行榜

定义：

- 每位玩家的 `totalScore = 所有局 settlement[playerId] 之和`

排序：

- 按 `totalScore` 从高到低排序
- 若同分，按 `winRate` 排
- 若仍相同，按 `dealerCount` 排

### 7.2 胜率

定义：

- `胜率 = 获胜局数 / 总参局数`

当前版本中每位玩家默认参与全部记录，因此：

- `总参局数 = 总局数`

胜负判定规则：

- 当前实现建议优先依据 `winnerPlayerIds` 统计胜负
- 对旧数据或普通抓分局，也可以从 `winnerSide + 角色结构` 推导 `winnerPlayerIds`

示例：

- A 是庄，C 是伴，B/D/E 是闲
- 若闲家抓分 `145`
- 则 `B/D/E` 本局记胜，`A/C` 本局记负

### 7.3 叫庄概率

由于当前版本只记录 `谁最终是庄`，不记录其他玩家是否曾尝试叫庄，因此这里的“叫庄概率”在产品中建议明确定义为：

- `叫庄概率（展示名可写为“叫庄率”） = 成为庄家的局数 / 总局数`

也就是统计 `最终坐庄频率`。

备注：

- 如果后续真的要计算“主动叫庄成功率”，则必须新增事件数据，例如 `谁翻 2`、`谁亮主`、`谁尝试过叫庄`
- 当前版本不具备这类原始数据，因此不能做“叫庄成功率”

### 7.4 其他可顺手统计的指标

虽然当前版本不一定首屏展示，但建议在数据层保留可计算能力：

- `dealerCount`：当庄次数
- `partnerCount`：当伴次数
- `idleCount`：当闲次数
- `bestRoundScore`：单局最高得分
- `worstRoundScore`：单局最低得分

## 8. 结算流程设计

### 8.1 单局保存流程

```text
1. 用户选择 `模式`
2. 用户选择 `结束方式`
3. 用户选择庄家
4. 系统根据模式联动展示输入项
5. 用户录入 `伴家 / 闲家抓分 / 逃跑玩家`
6. 系统做合法性校验
7. 根据 `mode + finishType` 选择对应结算逻辑
8. 生成 `settlement / winnerPlayerIds / ruleSnapshot`
9. 写入 rounds
10. 重算排行榜、总分、胜率、叫庄率
11. 刷新历史记录表
```

### 8.2 结算伪代码

```js
function settleRound(roundInput) {
  if (roundInput.finishType === "escape") {
    return settleEscapeRound(roundInput);
  }
  if (roundInput.mode === "solo") {
    return settleSoloRound(roundInput);
  }
  return settleNormalRound(roundInput);
}
```

### 8.3 当前实现优化点

为保证当前版本可扩展且兼容旧数据，已采用以下优化：

- `结算引擎拆分`
- `normal + score`、`solo + score`、`normal + escape`、`solo + escape` 分开处理
- `winnerPlayerIds` 统一胜负统计口径
- `旧数据兼容清洗`，避免历史记录因版本升级失效
- `表单联动显隐`，减少错误输入
- `历史记录和导出统一兼容多种局型`
