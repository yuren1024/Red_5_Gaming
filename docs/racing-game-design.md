# 2D 赛车游戏详细设计文档（V1）

## 1. 目标与范围

本文档用于定义 `Red_5_Gaming` 项目中第三个小游戏的当前版本设计方案。

当前版本计划新增一个 `俯视角 2D 单机赛车计时游戏`，强调：

- `方向键控制`
- `赛道内驾驶`
- `直线速度稳定`
- `转弯自动减速`
- `每圈计时`
- `本地最佳成绩保存`

当前版本必须支持：

- 独立页面运行
- 登录后从 `games.html` 进入
- 单人单车玩法
- 固定 1 张赛道
- 固定 1 种比赛模式
- 每圈记录单圈时间
- 显示当前圈、上一圈、最佳圈
- 本地保存最佳圈速
- 支持暂停、重开、返回游戏选择页

当前版本暂不包含：

- AI 对手
- 道具系统
- 多辆车选择
- 多赛道切换
- 联机
- 复杂物理模拟
- 车辆碰撞损坏
- 回放系统
- 排行榜联网同步

## 2. 产品定位

该游戏定位为一个 `轻量街机风格的本地计时赛车小游戏`。

它不是拟真赛车模拟器，也不是对抗型竞速游戏，而是一个强调：

- 入门简单
- 上手直观
- 可反复刷圈
- 适合浏览器静态页面实现

的计时挑战游戏。

与项目内现有游戏的关系如下：

- `红五五家`：偏工具和记录
- `贪吃蛇`：偏经典休闲反应类
- `赛车游戏`：偏操作与节奏控制类

因此它能形成较好的游戏类型补充。

## 3. 当前版本玩法定义

### 3.1 核心玩法

玩家驾驶一辆赛车在封闭赛道上完成若干圈挑战。

游戏过程围绕以下体验展开：

- 直道维持高速
- 入弯降速
- 控制方向避免出界
- 争取更快单圈时间

### 3.2 当前比赛模式

当前版本建议只提供一种模式：

- `3 圈计时赛`

设计原因：

- 有明确开始与结束
- 更像“一局完整游戏”
- 更适合展示单圈成绩和总成绩
- 便于后续扩展自由练习模式

每局流程为：

1. 玩家点击开始比赛
2. 进入 3 秒倒计时
3. 比赛开始，车辆从起点出发
4. 玩家连续完成 3 圈
5. 系统展示本局结果
6. 玩家可选择重新开始或返回

### 3.3 操作方式

键盘控制建议如下：

- `ArrowUp`：前进 / 保持动力
- `ArrowLeft`：左转
- `ArrowRight`：右转
- `ArrowDown`：轻刹车
- `Space`：暂停 / 继续

当前版本不强制支持 `W/A/S/D`，但可以预留兼容。

### 3.4 目标反馈

玩家在一局中需要关注三类结果：

- 当前圈是否顺利完成
- 当前圈速是否快于上一圈
- 当前圈速是否刷新本地最佳

## 5. 核心规则设计

### 5.1 比赛规则

当前版本采用固定 `3 圈计时赛`。

判定一局结束的条件为：

- 玩家完成第 3 圈有效圈
- 系统立即进入结果展示状态

有效圈需要满足：

- 车辆按正确方向通过起终点线
- 已按顺序通过本圈所需检查点
- 圈速高于最小有效圈时间阈值

最小有效圈时间建议设置为：

- `5 秒`

用途：

- 防止在起点附近误触发
- 防止倒车刷圈
- 防止异常判定

### 5.2 车辆运动规则

当前版本采用 `简化街机式运动模型`，而不是复杂物理模拟。

车辆包含以下核心属性：

- 位置 `x / y`
- 朝向角度 `heading`
- 当前速度 `speed`
- 目标速度 `targetSpeed`
- 当前是否在赛道内
- 当前是否处于弯道区域

### 5.3 速度规则

这是本游戏最关键的设计点。

建议使用三档速度体系：

- `直线速度`
- `转弯速度`
- `越界惩罚速度`

示例定义：

- 直线速度：`100%`
- 转弯速度：`70%`
- 越界惩罚速度：`45%`

规则如下：

1. 当车辆位于直线路段，且未发生明显转向时：
  - 目标速度为 `直线速度`
2. 当车辆进入弯道路段，或玩家持续按住左右方向键转向时：
  - 目标速度为 `转弯速度`
3. 当车辆压线或驶出赛道时：
  - 目标速度为 `越界惩罚速度`
4. 当前速度不立即跳变，而是以平滑方式逼近目标速度：
  - 入弯时较快减速
  - 出弯时较慢恢复

这样既符合“直线不变、转弯减速”的设想，也能让操作手感更自然。

### 5.4 转向规则

转向采用固定角速度模型：

- 按住 `←` 时朝左旋转
- 按住 `→` 时朝右旋转
- 车辆速度越低，转向越容易控制
- 车辆在高速状态下连续转向时，等效更容易走大弧线

建议转向规则：

- 只有在 `↑` 前进状态下才允许明显转向
- 未按前进键时，转向弱化或无效
- 刹车时保留少量转向修正能力

### 5.5 刹车规则

`↓` 的作用不是倒车，而是：

- 临时降低目标速度
- 帮助玩家更稳地过弯
- 便于在弯前主动修正路线

当前版本不建议支持倒车，原因：

- 会增加圈速判定复杂度
- 更容易出现起终点反复触发问题
- 对首版玩法价值不高

### 5.6 出界规则

当车辆中心点或关键采样点离开赛道宽度范围时，判定为出界。

出界后的处理建议分两级：

#### 轻度出界

条件：

- 刚刚离开赛道
- 仍靠近边缘
- 持续时间较短

处理：

- 立刻进入 `越界减速`
- 允许玩家自行回到赛道
- 车辆颜色或外发光提示变化

#### 严重出界

条件：

- 持续出界超过阈值
- 或距离赛道中心过远

处理：

- 强制重置到最近安全点
- 附加一个小时间惩罚，例如 `+1.5 秒`
- 显示提示：`偏离赛道，已重置`

当前版本建议保留严重出界自动重置机制，以避免车辆永久卡死在场外。

## 6. 赛道设计

### 6.1 当前版本赛道目标

当前版本只制作 1 张赛道，名称建议为：

- `练习环道`

设计目标：

- 易理解
- 易上手
- 能明显体现直线与弯道节奏差异
- 适合首版调手感

### 6.2 赛道结构建议

建议赛道整体为封闭环形，包含：

- 2 段长直道
- 2 个大弯
- 1 段中等复杂度连接弯

这样既不会太单调，也不会过难。

### 6.3 赛道区段分类

赛道可按顺序拆成若干逻辑区段：

1. `起跑直线`
2. `右弯区`
3. `后直线`
4. `左弯区`
5. `收尾连接区`
6. 回到起终点

每个区段包含元信息：

- 区段类型：`straight` / `curve`
- 建议方向：`left` / `right` / `none`
- 区段长度
- 区段限速类型

### 6.4 赛道表现方式

建议赛道底层不使用图片资源，而使用 Canvas 绘制。

推荐方案：

- 使用一条 `中心线`
- 给中心线设置固定 `赛道宽度`
- 绘制出完整道路轮廓
- 起终点线与检查点另外叠加绘制

这样做的优点：

- 不依赖外部素材
- 易于调整赛道形状
- 后续扩展第二张赛道更容易
- 适合当前纯原生 JS 项目

### 6.5 检查点设计

建议设置 3 个检查点：

- `CP1`
- `CP2`
- `CP3`

规则：

- 玩家必须按顺序通过所有检查点
- 然后回到起终点，才算完成有效圈

检查点判定可采用：

- 赛道中的一条横向线段
- 或一段窄矩形区域

用途：

- 防止绕近路刷圈
- 防止逆向穿线
- 保证每圈完整有效

## 7. 圈速与计时设计

### 7.1 计时开始时机

比赛开始流程：

1. 点击 `开始比赛`
2. 显示 `3`
3. 显示 `2`
4. 显示 `1`
5. 显示 `GO`
6. 当前圈计时从 `GO` 时刻开始

### 7.2 当前圈计时

比赛中持续展示：

- 当前圈实时计时

显示格式建议统一为：

- `MM:SS.mmm`

示例：

- `00:12.483`

### 7.3 完成一圈后的处理

当完成有效圈时，系统执行：

1. 记录当前圈时间
2. 写入本局圈速列表
3. 更新上一圈时间
4. 如果更快则更新最佳圈
5. 播放简单提示
6. 开始下一圈计时

### 7.4 最佳圈逻辑

需要维护两类最佳圈：

#### A. 本局最佳圈

仅在当前 3 圈比赛中比较。

#### B. 本地历史最佳圈

保存在浏览器 `localStorage` 中。

刷新历史最佳时建议有明确反馈：

- 文案：`新纪录！`
- HUD 高亮
- 短暂闪烁动画

### 7.5 结果展示

完成 3 圈后，进入结果状态。

建议展示：

- 总用时
- 3 圈成绩
- 最快圈
- 是否刷新本地最佳圈
- `重新开始` 按钮
- `返回游戏选择` 按钮

## 8. 状态机设计

### 8.1 页面状态

建议定义以下状态：

- `IDLE`：未开始
- `COUNTDOWN`：倒计时
- `RUNNING`：比赛中
- `PAUSED`：暂停中
- `FINISHED`：比赛结束

### 8.2 状态说明

#### `IDLE`

初始状态。

特征：

- 车辆停在起点
- 不计时
- 可点击开始比赛

#### `COUNTDOWN`

开始比赛前的倒计时阶段。

特征：

- 显示 3, 2, 1, GO
- 锁定大部分操作
- 不记录圈速

#### `RUNNING`

正常比赛阶段。

特征：

- 车辆移动
- 实时计时
- 检查碰撞、出界、过线、圈速

#### `PAUSED`

玩家按 `Space` 后进入。

特征：

- 暂停计时
- 冻结更新
- 保留当前局面

#### `FINISHED`

完成 3 圈后进入。

特征：

- 停止计时
- 展示结果
- 可重新开始或退出

### 8.3 状态切换

```text
IDLE -> COUNTDOWN -> RUNNING -> FINISHED
                    RUNNING <-> PAUSED
FINISHED -> COUNTDOWN
```

## 9. 页面流程设计

### 9.1 进入页面流程

1. 校验登录状态
2. 读取当前用户名
3. 初始化画布
4. 加载赛道数据
5. 读取本地最佳成绩
6. 渲染初始页面

### 9.2 开始比赛流程

1. 点击开始比赛
2. 重置比赛数据
3. 车辆回到起点
4. 清空本局圈速
5. 进入倒计时
6. 倒计时结束后正式开始

### 9.3 比赛中循环流程

每帧主要执行：

1. 读取输入状态
2. 更新速度目标
3. 更新朝向
4. 更新位置
5. 判定是否出界
6. 判定是否经过检查点
7. 判定是否通过起终点
8. 更新时间显示
9. 重绘画面

### 9.4 完赛流程

1. 第 3 圈完成
2. 计算总用时
3. 比较本地历史最佳
4. 保存成绩
5. 显示结果面板

## 11. 本地存储设计

建议使用独立的存储键：

- `hwwj.racing.timeTrial.v1`

存储内容建议包括：

- 每张赛道最佳单圈
- 最近若干次比赛结果
- 是否开启音效
- 未来可扩展的默认设置

建议只保留最近 `10` 条结果，避免无限增长。

## 12. 渲染与技术设计

### 12.1 技术方案

延续当前项目架构，使用：

- `HTML`
- `CSS`
- `JavaScript`
- `Canvas 2D`
- `localStorage`

不引入任何第三方依赖。

### 12.2 主循环方式

建议使用 `requestAnimationFrame` 作为游戏主循环。

原因：

- 比 `setTimeout` 更适合连续动画
- 浏览器渲染节奏更自然
- 适合基于 `deltaTime` 的位移与计时更新

### 12.3 每帧更新逻辑

每帧分两部分：

#### A. update

负责：

- 处理输入
- 更新速度
- 更新方向
- 更新位置
- 检查赛道状态
- 检查圈速逻辑

#### B. render

负责：

- 画背景
- 画赛道
- 画赛车
- 画状态文本

### 12.4 渲染层级建议

建议绘制顺序如下：

1. 背景
2. 草地区域
3. 赛道路面
4. 起终点线
5. 检查点提示
6. 赛车
7. 特效提示
8. HUD 覆盖层

## 15. 参数建议

以下是第一版建议调参方向，不要求最终固定。

### 15.1 车辆参数

- 直线目标速度：`260 px/s`
- 弯道目标速度：`180 px/s`
- 越界目标速度：`110 px/s`
- 入弯减速响应：较快
- 出弯恢复响应：中等
- 最大转向角速度：`2.4 rad/s`

### 15.2 比赛参数

- 倒计时：`3 秒`
- 圈数：`3`
- 最小有效圈时间：`5000 ms`
- 严重出界自动重置阈值：`1200 ms`
- 自动重置罚时：`1500 ms`

### 15.3 视觉参数

- 画布建议尺寸：`960 x 540`
- 赛道宽度：`64 ~ 80 px`
- 赛车尺寸：`28 x 16 px`

## 17. 当前版本边界说明

为了控制首版复杂度，当前版本刻意做以下限制：

- 只做俯视角，不做伪 3D
- 只做 1 张赛道
- 只做 1 辆车
- 只做单人计时，不做竞争
- 不做倒车
- 不做漂移和复杂抓地力模型
- 不做伤害系统
- 不做复杂音效系统
- 不做移动端触控首发支持

这些限制有利于把核心体验先做扎实，即：

- 操作是否顺手
- 赛道是否清晰
- 圈速判定是否稳定
- 刷最佳圈是否有重复游玩价值

## 19. MVP 开发优先级建议

如果后续进入实现，推荐按以下顺序推进：

1. 先完成页面框架、Canvas、HUD 和按钮布局
2. 再完成赛车移动、转向和赛道边界判定
3. 然后完成弯道减速与速度恢复
4. 接着完成检查点与单圈计时
5. 最后补本地最佳成绩、结果面板和视觉优化

## 20. 结论

该方案适合作为当前项目的第三个小游戏，原因是：

- 符合现有静态网页架构
- 能沿用独立页面 + 独立脚本的组织方式
- 不依赖后端或资源管线
- 玩法明确，首版边界清楚
- 有后续扩展空间

首版最重要的成功标准不是“物理是否真实”，而是：

- 玩家能顺畅跑起来
- 直线与弯道节奏差异明显
- 单圈计时准确可靠
- 刷最佳圈有成就感
